## 2025-02-14 - ResponsiveConfirmDialog for Destructive Actions
**Learning:** Destructive actions (like Delete) implemented with custom hardcoded modals lack standard accessibility attributes (`role="dialog"`, `aria-modal`, etc.) and mobile responsiveness (like bottom sheets). This app has a `ResponsiveConfirmDialog` component designed specifically for this purpose, but it was not being utilized uniformly.
**Action:** Always use `ResponsiveConfirmDialog` for destructive confirmation prompts (such as `DeleteResumeModal`) to ensure a consistent, accessible, and mobile-friendly UX that prevents accidental data loss.

## 2024-04-08 - Added explicit focus indicators to generic custom dialog buttons
**Learning:** Generic, frequently-used UI components like `ResponsiveConfirmDialog` act as foundational blocks for many specific modals (e.g., DeleteResumeModal). When these generic dialog components lack explicit `focus-visible` styling on primary interactable elements (like Close, Cancel, and Confirm buttons), keyboard navigation accessibility issues propagate broadly across the app. Standard `hover` or `transition-colors` without `focus-visible:ring-` often fall short of meeting accessibility standards for visually highlighting focused elements, especially on diverse backgrounds.
**Action:** Always ensure that structural UI primitives (like base modals, dialogs, standard button wrappers) provide robust, out-of-the-box keyboard focus states using Tailwind utilities (`focus:outline-none focus-visible:ring-2 focus-visible:ring-accent`, and `focus-visible:ring-offset-2`). Validate these base components rigorously before applying them to downstream product views.
